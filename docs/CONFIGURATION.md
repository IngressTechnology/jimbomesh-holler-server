# Configuration Reference

## Overview

Configuration is managed through a single `.env` file alongside `docker-compose.yml`.
On first boot, environment variables are seeded into a SQLite database. After that,
most settings can be edited live through the **Admin UI > Configuration** tab without
restarting the container. Port and TLS changes still require a restart.

| File | Purpose |
|------|---------|
| `.env` | Initial configuration (seeds SQLite on first boot) |
| `.env.example` | Documented template with defaults |
| `docker-compose.yml` | Service definitions (profiles, volumes, health checks) |
| `holler.db` (SQLite) | Persisted runtime settings (editable via Admin UI) |

### Precedence

1. **Admin UI edit** — saved to SQLite, takes effect within 5 seconds
2. **SQLite seed value** — written from `.env` on first boot only
3. **Environment variable** — used when SQLite has no value (fallback)
4. **Hard-coded default** — built into the gateway code

## Environment Variables

### API Authentication

| Variable | Default | Description |
|----------|---------|-------------|
| `JIMBOMESH_HOLLER_API_KEY` | (none) | **Required** — API key for Ollama access. Auto-generated by installers, or generate manually with `openssl rand -hex 32` |
| `GATEWAY_PORT` | `1920` | Internal gateway listen port inside the container (validated/authenticated API surface) |
| `OLLAMA_INTERNAL_PORT` | `11435` | Internal Ollama port (localhost only, not exposed) |
| `RATE_LIMIT_PER_MIN` | `60` | Maximum requests per minute per IP address |
| `ADMIN_ENABLED` | `true` | Enable admin UI at `/admin`. Set to `false` to disable |
| `ADMIN_API_KEY` | (none) | Separate API key for admin routes. Falls back to `JIMBOMESH_HOLLER_API_KEY` if not set |
| `RATE_LIMIT_BURST` | `10` | Allow short bursts above the per-minute rate limit |
| `SHUTDOWN_TIMEOUT_MS` | `10000` | Graceful shutdown drain timeout in milliseconds |
| `HOLLER_SERVER_NAME` | `Holler Server <hostname>` | Branded server name shown in Admin UI/login. `setup.sh` / `setup.ps1` auto-generates this when missing |

**Important:** `JIMBOMESH_HOLLER_API_KEY` is required. If it is missing, the gateway exits at startup instead of running without authentication.

### Runtime API Key Rotation

The API key can be changed at runtime through the **Admin UI > Configuration** tab without restarting the container:

1. Navigate to Configuration > Security
2. Click **Regenerate Key** and type `hellyeah` to confirm
3. The new key takes effect immediately (saved to SQLite as `api_key_override`)
4. Update your `.env` file to match the new key for persistence across container recreations

The runtime key override is stored in the SQLite `settings` table. On boot, the gateway checks for an `api_key_override` setting and uses it if present, falling back to the `.env` value.

| Variable | Default | Description |
|----------|---------|-------------|

### Enhanced Security (Bearer Tokens — Tier 2)

| Variable | Default | Description |
|----------|---------|-------------|
| `ENHANCED_SECURITY_ENABLED` | `false` | Enable Tier 2 bearer tokens (`jmh_*`) with scoped permissions, per-token rate limits, expiry, and usage tracking. Can also be toggled via Admin UI > Configuration > Security |

When enabled, clients can authenticate with `Authorization: Bearer jmh_...` tokens in addition to the standard `X-API-Key`. Tokens are managed via Admin UI or `POST/GET/DELETE/PATCH /admin/api/tokens`. Token hashes are stored in `/data/keys.json` on the `holler_data` volume. Admin routes continue to use the admin API key flow.

### Tier 3 JWT Validation (Mesh-Connected Mode)

Tier 3 JWT auth is not controlled by a single env var. It becomes active only when all of the following are true:

1. `JIMBOMESH_API_KEY` is set
2. `data/auth0-config.json` exists
3. `auth0-config.json` contains at least `domain` and `audience`

Example `data/auth0-config.json`:

```json
{
  "domain": "your-tenant.auth0.com",
  "audience": "https://api.jimbomesh.ai/",
  "issuer": "https://your-tenant.auth0.com/"
}
```

When configured, the gateway accepts `Authorization: Bearer <jwt>` on inference routes, validates RS256 signatures through JWKS, and applies per-buyer rate limits from JWT claims.

### Mesh Connectivity (Optional)

Used when connecting this Holler to the JimboMesh coordinator. Leave unset for standalone off-grid mode.

| Variable | Default | Description |
|----------|---------|-------------|
| `JIMBOMESH_API_KEY` | (optional) | Mesh API key used for coordinator registration, heartbeats, and job polling |
| `JIMBOMESH_COORDINATOR_URL` | (none) | Preferred coordinator URL override. If set, takes precedence over `JIMBOMESH_MESH_URL` |
| `JIMBOMESH_MESH_URL` | `https://api.jimbomesh.ai` | Legacy coordinator URL variable kept for backward compatibility |
| `JIMBOMESH_HOLLER_NAME` | (none) | Friendly Holler name shown in Mesh dashboards (falls back to `HOLLER_SERVER_NAME`, then sanitized `Holler-<hostname>` when unset) |
| `JIMBOMESH_AUTO_CONNECT` | `true` | Auto-connect to Mesh on startup when `JIMBOMESH_API_KEY` exists. Set `false` for manual connect from Admin UI |
| `MAX_PEER_CONNECTIONS` | `10` | Maximum concurrent WebRTC peer sessions for Mesh jobs. Set `0` to disable WebRTC and force HTTP polling |

Mesh status responses also include resilience telemetry:

- `httpPollingFallback`: `true` when the connector has temporarily fallen back to HTTP polling after repeated unstable management WebSocket reconnects.
- `fullTeardownAttempts`: count of full teardown/re-registration attempts made during recovery before entering fallback mode.

### Installer Persistence Behavior

The setup scripts persist all interactive choices into `.env` so reinstall/rebuild flows stay seamless:

- Preserve existing `.env` when present (no overwrite)
- Auto-generate `HOLLER_SERVER_NAME=Holler Server <hostname>` when missing/empty
- Persist deployment mode (`COMPOSE_FILE`, including GPU/mac overlays)
- Persist startup behavior and ports (`HOLLER_PERFORMANCE_MODE`, `GATEWAY_PORT`, `OLLAMA_HOST_PORT`)
- Persist model and mesh choices (`HOLLER_MODELS`, `OLLAMA_EMBED_MODEL`, `JIMBOMESH_AUTO_CONNECT`, `JIMBOMESH_HOLLER_NAME`)

Mesh settings can also be saved at runtime from the Admin UI Mesh panel (`/admin/api/mesh/settings`) and are persisted in SQLite.

**Mesh auth behavior:**
- Mesh connector requests always authenticate with `X-API-Key: <JIMBOMESH_API_KEY>`.
- The connector does not switch to Bearer auth after registration.
- Registration response tokens are ignored; the original mesh API key remains in use for register/heartbeat/job polling/WebRTC metadata calls.
- `JIMBOMESH_HOLLER_API_KEY` is separate local auth for this server's gateway/admin routes and is never mutated by mesh registration.

|----------|---------|-------------|
| `DOCUMENTS_COLLECTION` | `documents` | Default Qdrant collection name for document storage |

```bash
openssl rand -hex 32
```

## Installation Configuration

### `.setup-config.json`

`setup.sh` writes a `.setup-config.json` file in the project root after a successful installation. It is not an environment variable file — it is metadata for tooling and future installer runs.

```json
{
  "ollamaMode": "native",
  "installedAt": "2026-02-27T23:57:12Z",
  "securityWarningAccepted": true,
  "platform": "darwin",
  "arch": "arm64"
}
```

| Field | Values | Description |
|-------|--------|-------------|
| `ollamaMode` | `"native"` / `"docker"` | `"native"` = Performance Mode (Homebrew Ollama); `"docker"` = Secure Mode |
| `installedAt` | ISO 8601 timestamp | When the installer last ran |
| `securityWarningAccepted` | `true` / `false` | Whether the user accepted the Performance Mode security warning |
| `platform` | `"darwin"` / `"linux"` / `"win32"` | Host operating system |
| `arch` | `"arm64"` / `"x86_64"` | CPU architecture |

This file is listed in `.gitignore` — do not commit it to version control.

## Model Configuration

### Default Models

| Model | Type | Dimensions | Size | Purpose |
|-------|------|-----------|------|---------|
| `nomic-embed-text` | Embedding | 768 | ~274 MB | Text embeddings for Qdrant |
| `llama3.2:1b` | LLM | — | ~1.3 GB | Default lightweight local inference model |

### Alternative Embedding Models

| Model | Dimensions | Size | Notes |
|-------|-----------|------|-------|
| `mxbai-embed-large` | 1024 | ~670 MB | Higher quality, larger |
| `all-minilm` | 384 | ~45 MB | Tiny, fast, lower quality |
| `snowflake-arctic-embed` | 1024 | ~670 MB | Strong retrieval performance |
| `bge-large` | 1024 | ~670 MB | Good multilingual support |

See [MODEL_BENCHMARKS.md](MODEL_BENCHMARKS.md) for detailed latency benchmarks and quality comparisons to help you choose.

### Changing Models

1. Edit `.env`:

```bash
HOLLER_MODELS=mxbai-embed-large,llama3.2:3b
OLLAMA_EMBED_MODEL=mxbai-embed-large
EMBED_DIMENSIONS=1024
```

2. Restart to pull the new model:

```bash
docker compose up -d --force-recreate
```

> **Warning:** Changing embedding dimensions requires re-creating Qdrant collections and re-embedding all content. You cannot mix dimensions in the same collection.

## Docker Compose Configuration

| Mode | How to Enable | Services |
|------|--------------|----------|
| CPU (default) | `docker compose up -d` | `jimbomesh-still` |
| NVIDIA GPU | Set `COMPOSE_FILE=docker-compose.yml:docker-compose.gpu.yml` in `.env` | `jimbomesh-still` (with GPU) |
| macOS Metal GPU | Run `./setup.sh`, select Performance Mode; sets `COMPOSE_FILE=docker-compose.yml:docker-compose.mac.yml` | `jimbomesh-still` (gateway only; Ollama on host) |
| Qdrant | `--profile qdrant` | `jimbomesh-qdrant`, `init-qdrant` |

Both GPU and macOS modes use compose override files loaded via the `COMPOSE_FILE` environment variable. This ensures `docker compose down` always works without extra flags.

**macOS Performance Mode** requires `docker-compose.mac.yml` (committed to the repo; `setup.sh` overwrites it during Performance Mode setup). It sets `OLLAMA_EXTERNAL_URL` so the container gateway routes to the native Ollama on your Mac instead of starting its own. See [MAC_WINDOWS_SETUP.md](MAC_WINDOWS_SETUP.md).

GPU + Qdrant example:

```bash
# .env has COMPOSE_FILE set for GPU
docker compose --profile qdrant up -d
```

## Dimension Compatibility

JimboMesh production uses OpenAI `text-embedding-3-small` via OpenRouter, which produces **1536-dimensional** vectors. The default Ollama model `nomic-embed-text` produces **768-dimensional** vectors.

**You cannot mix dimensions in the same Qdrant collection.**

| Approach | Pros | Cons |
|----------|------|------|
| Fresh re-embed (recommended) | Clean, consistent | Must re-process all content |
| Match dimensions with different model | No re-embedding | Few local models match 1536d |
| Separate collections | Gradual migration | Requires JimboMesh code changes |

To do a fresh re-embed after switching to local Ollama:

```bash
# Re-create collections with new dimensions
docker compose --profile qdrant down -v
docker compose --profile qdrant up -d

# Trigger full re-sync in JimboMesh
docker exec jimbomesh-gateway /opt/jimbomesh/scripts/ingest-notion.sh --full
docker exec jimbomesh-gateway /opt/jimbomesh/scripts/ingest-hubspot.sh --full
```
